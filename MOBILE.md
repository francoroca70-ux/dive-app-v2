# Seven Seas — Mobile App (Capacitor)

The native iOS/Android wrapper around the existing web app. Same code, same Supabase
backend — Capacitor just packages `index.html` + friends into something installable
from an app store instead of a browser.

## How it fits together

- `www/` — a clean copy of the deployable web app (`index.html`, `sw.js`,
  `manifest.json`, `assets/`), rebuilt from the real files any time you run
  `npm run build:www`. This is what actually gets bundled into the native apps —
  keeps `supabase/`, `legal/`, `.md` docs, etc. out of the shipped app.
- `android/` — the native Android Studio project (generated by Capacitor).
- `ios/` — the native Xcode project (generated by Capacitor).
- `capacitor.config.json` — app ID (`com.sevenseasops.app`), app name ("Seven Seas"),
  brand colors.
- `resources/` — source icon/splash images (`icon.png`, `splash.png`) plus the
  reusable white-trident-on-transparent master used to generate every icon size for
  both platforms.
- `.github/workflows/mobile-build.yml` — builds both apps in the cloud automatically.

## Building — no local Android Studio or Mac needed

GitHub Actions builds both apps for you, using the GitHub account/repo you already
have (no new signups for this part):

1. Go to the repo on GitHub → **Actions** tab → **Mobile app builds (Android + iOS)**
   → **Run workflow**.
2. Android finishes first (a few minutes) — download the `seven-seas-android-debug`
   artifact, it's a real `.apk` you can sideload onto an Android phone to test
   (Settings → allow installs from unknown sources, then open the file).
3. iOS currently builds for the **Simulator only** (proves the app compiles) since
   there's no Apple Developer account/certificates set up yet — see
   `App-Store-Roadmap.md` for that next step. Once that's in place, this workflow
   gets extended to produce a real signed build installable on an actual iPhone via
   TestFlight.

It also runs automatically on every push to `main` that touches the app, `android/`,
`ios/`, or icon/config files — so it stays current without having to remember to
trigger it.

## If you ever want to open the native projects yourself

You don't need to for day-to-day work — the CI pipeline above handles builds. But if
you (or I, working alongside you on a call) ever want to open them directly:

- Android: open the `android/` folder in Android Studio.
- iOS: needs a Mac — open `ios/App/App.xcworkspace` (not `.xcodeproj`) in Xcode.
